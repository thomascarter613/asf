# Clean-Room Pattern Adoption Policy

## 1. Purpose

This policy defines how the Agentic Software Foundry may study, learn from, reference, and independently implement architectural ideas from external systems while protecting the project from legal, ethical, security, and provenance risks.

The Foundry is intended to learn from public architecture patterns, open-source projects, public documentation, public research, public discussions, and the project owner’s own prior work.

The Foundry must not incorporate leaked proprietary source code, copied private implementation details, unauthorized material, or derivative implementations of non-open-source code.

This policy exists to preserve the independence, defensibility, auditability, and trustworthiness of the Foundry.

## 2. Controlling Principle

The Foundry may learn from public ideas.

The Foundry must not copy protected implementation.

The distinction is:

```text
Allowed:
public concepts, public architecture patterns, public documentation,
open-source code used according to its license, independently authored designs,
first-principles reasoning, and the project owner’s own prior work.

Forbidden:
leaked proprietary source code, copied private implementation details,
translated proprietary code, copied tests, copied comments, copied prompts,
copied internal file structures, unauthorized secrets, or derivative works
based on non-open-source material.
````

## 3. Scope

This policy applies to:

1. source code;
2. documentation;
3. prompts;
4. agent role definitions;
5. tests;
6. templates;
7. configuration files;
8. schemas;
9. architectural diagrams;
10. implementation plans;
11. workflows;
12. examples;
13. generated artifacts;
14. comments;
15. naming conventions;
16. runtime behavior;
17. any other project artifact.

This policy applies whether the material is written by a human, generated by an AI assistant, copied from a public source, produced from memory, derived from notes, or created through automated tooling.

## 4. Sources the Foundry May Study

The Foundry may study the following categories of sources.

### 4.1 Public Documentation

The Foundry may study official public documentation, public API references, public architecture guides, public blog posts, public talks, public issue discussions, public standards, and public tutorials.

Public documentation may influence architecture, terminology, trade-off analysis, and implementation strategy.

### 4.2 Open-Source Projects

The Foundry may study open-source projects when their licenses permit review, use, or adaptation.

When open-source code is reused directly, the applicable license must be respected.

When only architecture patterns are studied, the resulting implementation must be independently authored unless direct reuse is explicitly intended, licensed, and documented.

### 4.3 Public Architecture Patterns

The Foundry may adopt general architecture patterns such as:

1. event routers;
2. command buses;
3. append-only logs;
4. vector indexes;
5. repo-backed memory;
6. task runners;
7. multi-agent role separation;
8. worktree isolation;
9. workflow engines;
10. context-pack assembly;
11. notification sinks;
12. policy gates;
13. manifest-driven configuration;
14. lifecycle state machines;
15. verification loops.

General patterns are not owned by any one tool or vendor.

### 4.4 Prior Independent Project Work

The Foundry may reuse, refine, migrate, or supersede architecture developed in the project owner’s prior projects, including but not limited to Astra-Code and earlier Charon, AI SDLC, and Monorepo Factory design work.

When prior project material is incorporated, the Foundry should preserve provenance where practical.

### 4.5 First-Principles Reasoning

The Foundry may design systems from first principles based on project goals, constraints, engineering trade-offs, safety requirements, and known software engineering practices.

First-principles reasoning is preferred when external inspiration is unnecessary or creates avoidable ambiguity.

## 5. Sources the Foundry Must Not Use

The Foundry must not use the following.

### 5.1 Leaked Proprietary Source Code

The Foundry must not incorporate, translate, port, rewrite, or adapt leaked proprietary source code.

This includes leaked source maps, accidentally published proprietary packages, private repository contents, private prompt catalogs, private internal tests, or any other non-open material made public through error, breach, misconfiguration, or unauthorized disclosure.

### 5.2 Non-Public Implementation Details

The Foundry must not use private implementation details that are not part of public documentation or authorized open-source releases.

This includes:

1. copied function bodies;
2. copied class structures;
3. copied internal prompts;
4. copied private file layouts;
5. copied proprietary tests;
6. copied comments;
7. copied internal schemas;
8. copied internal workflows;
9. copied agent instructions;
10. copied hidden configuration.

### 5.3 Unauthorized Secrets or Credentials

The Foundry must not use, store, test, share, or preserve unauthorized secrets, tokens, keys, credentials, private endpoints, internal hostnames, customer data, or private operational details.

If any such material is encountered, it must be ignored, removed, and not committed.

### 5.4 Malware-Lure Repositories

The Foundry must not download, run, fork, vendor, or inspect suspicious repositories claiming to contain leaked code, unauthorized tool sources, cracked tools, private agent systems, or unofficial mirrors of proprietary systems.

### 5.5 License-Incompatible Material

The Foundry must not incorporate open-source code or documentation in a way that violates its license.

When there is doubt, the project should prefer independent implementation or avoid direct reuse.

## 6. Pattern Adoption Rules

The Foundry may adopt external ideas only through clean pattern adoption.

A clean pattern adoption must satisfy the following rules.

### 6.1 Use the Public Pattern, Not the Private Implementation

The Foundry may say:

```text
A daemon-first event router is useful.
A typed event model is useful.
A renderer/sink split is useful.
A directive-to-work-packet compiler is useful.
A planner/executor/reviewer separation is useful.
```

The Foundry must not say:

```text
Copy this project’s implementation.
Translate this private code into another language.
Recreate this private module structure.
Copy this prompt exactly.
Copy this test suite exactly.
```

### 6.2 Rename and Reframe Where Appropriate

The Foundry should use its own terminology where doing so improves clarity and avoids accidental imitation.

External project names may be referenced in research notes, acknowledgments, or comparative architecture documents, but internal Foundry components should be named according to the Foundry architecture.

### 6.3 Preserve Responsibility Boundaries

External patterns must be integrated into the proper layer.

Examples:

1. directive-to-workflow patterns belong in the AI SDLC Framework and Directive Compiler;
2. event routing patterns belong in the Foundry Event Bus and Notification Router;
3. multi-agent coordination patterns belong in the Foundry Agent Runtime;
4. repo-backed memory patterns belong in Charon;
5. template generation patterns belong in the Monorepo Factory;
6. vector retrieval patterns belong in Argos and Hephaestus.

External patterns must not collapse the Foundry’s core responsibility boundaries.

### 6.4 Prefer Independent Schemas

When possible, Foundry manifests, event schemas, work-packet schemas, context-pack schemas, and memory schemas should be independently designed around Foundry requirements.

They may be inspired by common industry practices, but they should not copy another project’s private schema.

### 6.5 Preserve Provenance

When a public source influences an architecture decision, the decision document should acknowledge the source category.

Example:

```text
This ADR is informed by public event-router patterns, public multi-agent
orchestration patterns, and the project owner’s prior Astra-Code work.
```

The project does not need to cite every general idea, but material inspirations should be documented.

## 8. Clean-Room Design Workflow

When adopting an external pattern, the Foundry should use the following workflow.

### 8.1 Describe the Need

First, describe the Foundry problem being solved.

Example:

```text
The Foundry needs event routing outside the coding agent context window
so that agents can focus on implementation while operators still receive
clear status updates.
```

### 8.2 Describe the General Pattern

Next, describe the general architecture pattern without copying implementation.

Example:

```text
Use typed events, event sources, an append-only log, a router, renderers,
and sinks.
```

### 8.3 Define the Foundry-Specific Design

Then define the Foundry implementation in Foundry terms.

Example:

```text
The Foundry Event Bus writes JSONL events to .foundry/events/events.jsonl.
The Notification Router reads events, renders status messages, and writes
to console and filesystem sinks in v1.0.
```

### 8.4 Implement Independently

Write the implementation from scratch using Foundry requirements, Foundry schemas, and Foundry tests.

### 8.5 Document Provenance

If the pattern was materially inspired by a named public project, document the influence in an ADR or architecture note.

### 8.6 Verify License and Policy Compliance

Before committing, confirm that no forbidden material has entered the repository.

## 9. AI Assistant Rules

AI assistants working on the Foundry must obey this policy.

An AI assistant must not:

1. provide leaked proprietary source code;
2. translate leaked proprietary source code;
3. reconstruct private implementation details;
4. produce code based on unauthorized material;
5. paste copied proprietary prompts;
6. create files derived from private tests or private comments;
7. claim that leaked material is open source;
8. hide external provenance;
9. silently introduce license-incompatible material;
10. bypass clean-room restrictions for convenience.

An AI assistant may:

1. explain public architecture patterns;
2. compare systems at a high level;
3. propose clean independent implementations;
4. write original code;
5. write original documentation;
6. create original tests;
7. cite or acknowledge public sources in documentation where appropriate;
8. help design Foundry-native patterns.

## 10. Repository Hygiene Rules

The repository must not contain:

1. leaked source code;
2. leaked source maps;
3. unauthorized secrets;
4. private internal prompts from third parties;
5. copied proprietary tests;
6. copied proprietary comments;
7. suspicious mirrored leak repositories;
8. vendor drops of unclear origin;
9. generated artifacts whose provenance is unknown;
10. unreviewed AI output claiming to reproduce private tools.

If such material is accidentally introduced, it must be removed immediately and the event must be documented.

## 11. License Handling

When direct reuse of open-source material is considered, the project must record:

1. source project name;
2. source URL;
3. license;
4. files or portions reused;
5. reason for reuse;
6. compatibility assessment;
7. attribution requirements;
8. whether the material is vendored, adapted, or used as a dependency.

For v1.0, the preferred approach is independent implementation rather than direct reuse of external code.

## 12. Security Handling

The Foundry must treat suspicious leak-related repositories, unofficial mirrors, and unauthorized code dumps as security risks.

The project should not download or execute them.

If public reporting describes malicious repositories or dependency-confusion attempts around a leaked tool, the Foundry should avoid those repositories entirely and use only trustworthy public sources, official documentation, and legitimate open-source repositories.

## 13. Documentation Requirements

When an architectural decision materially adopts an external public pattern, the relevant ADR should include a short note such as:

```text
Pattern note:
This decision is informed by public examples of event-router architectures,
multi-agent orchestration systems, workflow wrappers, and the project
owner’s prior Astra-Code research. The implementation is original and
Foundry-specific.
```

The note should not imply endorsement by or affiliation with the external project.

## 14. Naming and Branding Rules

The Foundry must not use names, marks, logos, or branding from external projects in a way that implies affiliation, sponsorship, endorsement, or derivation.

External names may be used only for comparison, research notes, or attribution.

Internal Foundry components should use Foundry-native names and terminology.

## 15. Enforcement

This policy is enforced through:

1. human review;
2. project discipline;
3. documentation review;
4. repository hygiene;
5. future repo contract tests;
6. future policy checks;
7. clean commit history;
8. explicit provenance notes;
9. refusal to incorporate suspicious material;
10. removal of non-compliant artifacts.

A future version of the Foundry should include automated policy checks for forbidden files, suspicious provenance, missing licenses, and required attribution.

## 16. v1.0 Policy Position

For v1.0, the Foundry will:

1. implement all core code independently;
2. use public architecture patterns only as inspiration;
3. avoid direct reuse of external project code unless clearly justified;
4. avoid all leaked or unauthorized material;
5. document major external pattern influences;
6. keep the repository defensible and auditable.

## 17. Amendment

This policy may be amended only through an explicit documented decision.

Any amendment must preserve the central rule:

```text
The Foundry may learn from public ideas, but it must not copy protected
implementation.
```

## 18. Status

This policy is active from the beginning of the Agentic Software Foundry project.

All future documentation, architecture, implementation, scaffolding, and AI-assisted development must comply with it.
